    最早发于MT, 2019-08-12

    如何优雅地解决掉NPE？

    • Optional类的灵感来自于一些其他的FP语言，主要的意义就是消除imperative形式的null检查，以及非常丑陋的判空的卫语句。
    • Optional的API的设计思想和Stream有一定共性
    • Optional的本质就是用类型信息和声明式逻辑相结合的方法，力求替换掉null检查带来的数据流和控制流的结合
    • 注意事项
      • Stream的map、flatMap、filter方法，在Optional类上都有相似的执行逻辑，注意flatMap可以展开多层Optional嵌套
      • 不推荐使用基本类型的Optional类型：
        • 这里不再需要考虑性能
        • OptionalInt之类的类型没有map、flatMap、filter这些方法——其FP的表达能力被限制了

  • lambda表达式和方法引用

    2019-10-22 |
    • { mt }
    • { java }
    • { java8 }
    /
    • { tech }

    最早发于MT, 2019-08-12

    • lambda表达式的类型信息是残缺的
      • lambda相比匿名类更加简洁，但是其只有所谓的function descriptor，而没有匿名类的父类那样清晰的类型
        • 诸如 Object o = () -> {} 这样的语句是无法通过编译的，lambda表达式只能被赋给类型匹配的函数式接口
        • Object o = (Runnable) () -> {} 是可以的，lambda就差一个接口信息
        • 如果函数重载靠两个target type相同的函数式接口参数区分，那个上面的转换（其实是一种类型标注）就是必要的，否则编译器会混淆
